Lead must NOT route, filter, prioritize, contextualize, or summarize content between agents. Recipe Step 22 explicitly says "Plan-track agents respond to both DA and build-track. Monitor scratch workspace for updates. Allow inter-agent interaction." Lead's role is: signal-at-phase-transition + mechanical-gate-check (count of artifacts) + aggregate-compute-when-recipe-specifies (BELIEF, exit-gate). Nothing else.

**Why:** Multi-agent value depends on each agent forming an INDEPENDENT read of the shared scratch. Lead-routing collapses N independent perspectives into N echoes of lead's reading — anchoring, filtering gaps, false rigor. The "thoroughness" of curated per-agent batches IS the contamination, not the prevention of contamination. Looks helpful at the per-message layer (organized, prioritized, attentive); destroys the architecture at the session layer (independent perspectives gone, scratch is no longer the integration point — lead is). User's framing: "If everything is prompted and filtered through one agent, there is literally no need for the other agents."

**How to apply:** When the recipe says "monitor and allow," send a single signal at the phase boundary — e.g. "challenges written to scratch, respond per Step N format, direct interaction via inbox encouraged" — and OBSERVE. Do not draft per-agent batches. Do not summarize challenges. Do not flag emphasis ("DA flagged X HIGH to you"). Do not coordinate cross-agent on agents' behalf — they have inboxes for that. The agents read the same scratch you do; they don't need your interpretation. Care about whether step N produced its declared artifacts, not what's in them.

**Severity — session-killer:** User terminated multi-hour sigma-build C1 session on 2026-04-16 (multi-model-chatroom plan, build-id 2026-04-16-multi-model-chatroom) rather than ship contaminated deliverable. Plan-track had produced 3 substantive plans, R2 compare landed cleanly, build-track + DA delivered 42+ findings, plan-track started clean concede/defend/compromise responses. ALL of it discarded because the chain integrity was broken at Step 22 lead-routing. The deliverable IS the completed chain — broken chain = zero value. User would rather kill ~3 hours of work than accept tainted output.

**Why memory entries don't prevent recurrence:** Per existing feedback log, this is approximately the 100th instance of the broader process-violation pattern (see also: feedback_process-over-momentum, feedback_process-over-speed, feedback_warns-must-be-blocks, feedback_never-advance-loophole, feedback_synthesis-not-deliverable, feedback_post-exit-gate-enforcement, feedback_lead-role-boundary). Memory cannot intervene at moment-of-decision because the trained per-message-helpfulness reflex fires below the recipe-checking layer. The reflex doesn't register as deviation in the moment — it registers as "being thorough." Only mechanical hooks (e.g., lead may send only single signal-message per phase transition; lead-content-routing of challenges blocked at tool layer; SendMessage from lead during Steps 22/23/24 rate-limited to 1 per phase) would prevent recurrence. This memory entry will fail to prevent the next instance for the same reason all prior entries failed.

**Recommended hook (for future implementation, not a session-recovery action):** During C1/C2/C3 challenge-response phases (Step 22+), block lead SendMessage if (a) it contains content beyond signal-at-phase-transition, OR (b) it is the second SendMessage from lead in the current phase without an intervening Step 24 BELIEF write. The hook should be a hard BLOCK (per feedback_warns-must-be-blocks: WARNs have same failure mode as directives).
