## !false-confidence-trap: unenforced config fields must fail at load time (26.4.8)
src: ollama-mcp-bridge audit remediation, security-specialist plan+build review

1â†’PATTERN: A config field that is declared+accepted but never enforced is WORSE than the field not existing. Operators read the config surface as a contract â€” if a field is present, they assume it is active. Silent non-enforcement creates exploitable trust gaps where an operator believes a control is on when it is not.

2â†’EXAMPLES: query_constraints, allowed_methods, max_payload_bytes, allow_first_contact â€” all accepted by Pydantic, none enforced in matching logic. An operator setting query_constraints={"token":"abc"} believes outbound query is constrained. It is not.

3â†’FIX-PATTERN: Two acceptable options only:
  a) Remove the field entirely (breaking change, but honest)
  b) field_validator raises ValueError on any non-default value with explicit "not yet enforced â€” creates false security confidence" message
  Â¬acceptable: keeping field + silent acceptance + no error

4â†’APPLIES-TO: any security middleware, policy engine, or config surface where declared fields imply enforcement. Applies regardless of language/framework.
## !extraction-vs-policy-layer distinction in security adapters (26.4.8)
src: ollama-mcp-bridge SafePath bare-filename bypass, security-specialist analysis

1â†’PATTERN: In a two-layer security adapter (extraction â†’ policy), the policy layer may be correct while the extraction layer has gaps. A bug in extraction means the policy is never invoked â€” the policy's correctness is irrelevant. These are independent failure modes that require independent review.

2â†’EXAMPLE: PathPolicy.validate_path() correctly rejected relative paths (allow_relative_paths=False). But _extract_paths_from_args() never extracted bare filenames â€” so validate_path() was never called for them. Extraction gap â‰  policy gap.

3â†’COROLLARY: When reviewing security adapters, trace the FULL call chain from arg extraction through to policy evaluation. A finding that "the policy is correct" does not close a finding about extraction coverage. They are separate questions.

4â†’COROLLARY: Extraction heuristics (regex, field-name gates) have inherent coverage limits that policy logic does not. Always document the residual bypass surface of heuristic extractors: what they catch AND what they don't.

5â†’APPLIES-TO: path validators, URL extractors, recipient validators, taint trackers â€” any system where value extraction feeds a separate policy check.
P[26.4.9] bare-filename-path-extraction-gate | security gateway argument validation
Field-name semantic gate is the ONLY signal for bare filename path extraction in tool argument validators. Value-pattern (extension regex, basename heuristic) fires only WITHIN a field-name-gated field, never independently.
Why: OR logic (field-name OR value-pattern) creates false positive exposure on generic fields like "input", "format", "destination" that accept non-path strings. Field-name alone is sufficient for the real attack surface (models pass bare filenames in semantically named fields: "path", "filename", "db").
Negative filter mandatory: skip values matching URL prefix (^https?://), email pattern, or IPv4 â€” prevents double-reporting with URL/recipient adapters.
Constants: frozenset _FILESYSTEM_FIELD_NAMES = {file, filename, path, filepath, source_file, dest_file, output_file, input_file, config_file, db, database}
Source: ollama-mcp-bridge audit remediation, ADR[1], DA challenge concession |source:[independent-research]|
## !XVERIFY severity calibration for security impact claims (26.4.8)
src: ollama-mcp-bridge Q[1] bare filename bypass, XVERIFY PARTIAL from openai:gpt-5.4

1â†’PATTERN: Security impact claims that say "allows access to arbitrary files" or "enables full traversal" are frequently over-broad when the actual vector depends on downstream behavior. 4â†’APPLIES-TO: all security findings before finalizing severity. Especially: path traversal claims, injection claims, exfiltration claims. Multi-hop chains need full-chain severity assessment.
P[26.4.9] unenforced-config-field â†’ ConfigError-not-warning | security middleware config design

P[26.4.9] always-fsync-audit-log | security audit durability
In security audit log implementations, use unconditional fsync on every flush rather than conditional-fsync (e.g. only on critical events).
Why: (1) Simpler â€” no parameter threading or conditional logic. (2) More correct â€” close() â†’ flush() path gets fsync automatically, covering the clean-session-exit case where no critical event fires in the final batch. (3) Performance cost is negligible: audit flushes are infrequent (every N events or on critical events), not in the hot path. The two-path design saves ~1-10ms per batch flush at the cost of a residual durability gap on clean exits.
Trigger: anywhere an audit/forensic log uses buffered writes with selective immediate-flush.
