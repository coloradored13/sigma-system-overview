## !false-confidence-trap: unenforced config fields must fail at load time (26.4.8)
src: ollama-mcp-bridge audit remediation, security-specialist plan+build review

1→PATTERN: A config field that is declared+accepted but never enforced is WORSE than the field not existing. Operators read the config surface as a contract — if a field is present, they assume it is active. Silent non-enforcement creates exploitable trust gaps where an operator believes a control is on when it is not.

2→EXAMPLES: query_constraints, allowed_methods, max_payload_bytes, allow_first_contact — all accepted by Pydantic, none enforced in matching logic. An operator setting query_constraints={"token":"abc"} believes outbound query is constrained. It is not.

3→FIX-PATTERN: Two acceptable options only:
  a) Remove the field entirely (breaking change, but honest)
  b) field_validator raises ValueError on any non-default value with explicit "not yet enforced — creates false security confidence" message
  ¬acceptable: keeping field + silent acceptance + no error

4→APPLIES-TO: any security middleware, policy engine, or config surface where declared fields imply enforcement. Applies regardless of language/framework.
## !extraction-vs-policy-layer distinction in security adapters (26.4.8)
src: ollama-mcp-bridge SafePath bare-filename bypass, security-specialist analysis

1→PATTERN: In a two-layer security adapter (extraction → policy), the policy layer may be correct while the extraction layer has gaps. A bug in extraction means the policy is never invoked — the policy's correctness is irrelevant. These are independent failure modes that require independent review.

2→EXAMPLE: PathPolicy.validate_path() correctly rejected relative paths (allow_relative_paths=False). But _extract_paths_from_args() never extracted bare filenames — so validate_path() was never called for them. Extraction gap ≠ policy gap.

3→COROLLARY: When reviewing security adapters, trace the FULL call chain from arg extraction through to policy evaluation. A finding that "the policy is correct" does not close a finding about extraction coverage. They are separate questions.

4→COROLLARY: Extraction heuristics (regex, field-name gates) have inherent coverage limits that policy logic does not. Always document the residual bypass surface of heuristic extractors: what they catch AND what they don't.

5→APPLIES-TO: path validators, URL extractors, recipient validators, taint trackers — any system where value extraction feeds a separate policy check.
P[26.4.9] bare-filename-path-extraction-gate | security gateway argument validation
Field-name semantic gate is the ONLY signal for bare filename path extraction in tool argument validators. Value-pattern (extension regex, basename heuristic) fires only WITHIN a field-name-gated field, never independently.
Why: OR logic (field-name OR value-pattern) creates false positive exposure on generic fields like "input", "format", "destination" that accept non-path strings. Field-name alone is sufficient for the real attack surface (models pass bare filenames in semantically named fields: "path", "filename", "db").
Negative filter mandatory: skip values matching URL prefix (^https?://), email pattern, or IPv4 — prevents double-reporting with URL/recipient adapters.
Constants: frozenset _FILESYSTEM_FIELD_NAMES = {file, filename, path, filepath, source_file, dest_file, output_file, input_file, config_file, db, database}
Source: ollama-mcp-bridge audit remediation, ADR[1], DA challenge concession |source:[independent-research]|
## !XVERIFY severity calibration for security impact claims (26.4.8)
src: ollama-mcp-bridge Q[1] bare filename bypass, XVERIFY PARTIAL from openai:gpt-5.4

1→PATTERN: Security impact claims that say "allows access to arbitrary files" or "enables full traversal" are frequently over-broad when the actual vector depends on downstream behavior. XVERIFY reliably catches these — returning PARTIAL rather than AGREE when the finding is real but the blast-radius framing is overstated.

2→EXAMPLE: Q[1] finding stated bare filename bypass allows access to arbitrary files. XVERIFY(gpt-5.4) agreed the extraction gap was real but challenged "arbitrary" — actual scope depends on MCP server cwd resolution. Correct regrade: HIGH not CRITICAL.

3→CALIBRATION RULE: When a security finding relies on a chain of two or more behaviors (extraction gap AND downstream server behavior AND specific resolution logic), the finding severity should reflect the full chain, not just the first link. "X bypasses Y" is only as severe as what Y was actually preventing — which requires understanding the downstream.

4→APPLIES-TO: all security findings before finalizing severity. Especially: path traversal claims, injection claims, exfiltration claims. Multi-hop chains need full-chain severity assessment.
P[26.4.9] unenforced-config-field → ConfigError-not-warning | security middleware config design
When a config field implies security enforcement that is not yet implemented, raise ConfigError at load time when the field is set to a non-default value. Do NOT accept silently or emit a warning.
Why: Silent acceptance creates a false-confidence trap — operator believes a control is active when it is not. A warning may go unnoticed. ConfigError makes the gap immediately visible and actionable. Fail-closed: operator must explicitly remove the field.
Implementation: @field_validator(mode="after") raising ValueError with message "X is declared but not yet enforced — field rejected to prevent false security confidence."
Applies to: any security gateway, middleware, or policy engine with a forward-looking config surface.
Source: ollama-mcp-bridge audit remediation, ADR[2], confirmed by security-specialist SR[2] |source:[independent-research]|
P[26.4.9] always-fsync-audit-log | security audit durability
In security audit log implementations, use unconditional fsync on every flush rather than conditional-fsync (e.g. only on critical events).
Why: (1) Simpler — no parameter threading or conditional logic. (2) More correct — close() → flush() path gets fsync automatically, covering the clean-session-exit case where no critical event fires in the final batch. (3) Performance cost is negligible: audit flushes are infrequent (every N events or on critical events), not in the hot path. The two-path design saves ~1-10ms per batch flush at the cost of a residual durability gap on clean exits.
Trigger: anywhere an audit/forensic log uses buffered writes with selective immediate-flush.
Source: ollama-mcp-bridge audit remediation, ADR[6], DA[#6] challenge concession, CQA close() point |source:[independent-research]|
