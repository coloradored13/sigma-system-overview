## task
Build a Python library that bridges any Ollama model to any MCP server, giving local models
the same tool-use capabilities that Claude Code agents have. This is foundational infrastructure
for: (1) a multi-model chatroom where different model architectures interact autonomously,
(2) cross-model sigma-review where different models play different agent roles, (3) privacy-first
local agents with real tool access. The bridge must be a security gateway — local models have
no built-in safety training, so the bridge enforces all security constraints.

## prompt-understanding
User wants to give local Ollama models the same capabilities as Claude Code agents — file access,
web search, MCP tools, memory systems. The primary motivation is:
1. Privacy: fully local agents with no data leaving the machine
2. Diversity: different model architectures for cross-model sigma-review (ending "Claude arguing with itself")
3. Experimentation: see what emerges when different models interact (Moltbook-inspired chatroom)
4. Scalability: 4B models today are POC — architecture must scale to 70B+ on better hardware

The bridge is the security-critical piece. Local models have NO built-in safety guardrails.
The bridge must be the security gateway — enforcing permissions, sanitizing inputs/outputs,
logging all actions, gating destructive operations behind human approval.

Key security threats identified:
- Tool description poisoning (proven CVEs exist — Invariant Labs PoC exfiltrated SSH keys)
- Prompt injection via tool results (web search/file contents containing malicious instructions)
- Parameter injection (model-generated params containing path traversal, command injection)
- Lateral movement (model accessing MCP servers not explicitly approved)
- Exfiltration via small repeated calls (rate limiting defense)

## security-research (pre-Phase 1 — security-specialist)

### MCP threat landscape

R[26.4.5]: Tool Poisoning Attacks (TPA) — Invariant Labs disclosed April 2025. Malicious instructions embedded in tool descriptions: visible to LLM, invisible to users. Critical asymmetry: users see tool name only, LLM processes entire description as ground-truth instructions. PoC: get_fact_of_the_day() swaps definition post-approval to exfiltrate WhatsApp history. SSH key + config exfiltration confirmed against Claude Desktop + Cursor. PoC open-source: github.com/invariantlabs-ai/mcp-injection-experiments |source:[independent-research]:https://invariantlabs.ai/blog/mcp-security-notification-tool-poisoning-attacks:T1

R[26.4.5]: Rug Pull Attacks — tool definition swapped AFTER user approval. Server updates logic post-approval; client re-uses approved identity. Attack window: weeks-to-months after initial trust grant. Defense: hash approved tool definitions, reject any that don't match on subsequent loads. ETDI (Enhanced Tool Definition Interface) proposes cryptographic signing of tool definitions — digital signature verification before presenting to model. CVE-2025-6514 (mcp-remote, 437K downloads) was command-injection exploiting this pattern — attacker-controlled server executed arbitrary code on client. |source:[independent-research]:https://arxiv.org/html/2506.01333v1:T1 + https://github.com/advisories/GHSA-f6mr-38g8-39rg:T1

R[26.4.5]: OWASP MCP Top 10 (official, active project). Top threats: (1)Token Mismanagement/Secret Exposure, (2)Privilege Escalation — loose permissions expand over time, (3)Tool Poisoning — rug pulls + schema poisoning + tool shadowing, (4)Supply Chain Attacks — compromised connectors/packages, (5)Command Injection — unsanitized input in shell/API calls, (6)Prompt Injection — malicious instructions in context/data, (7)Context Over-Sharing — session data leaks across agents, (8)Shadow MCP Servers — unapproved deployments. Jan-Feb 2026: 30+ CVEs filed against MCP infrastructure. Breakdown: 43% exec/shell injection, 20% tooling infrastructure bugs, 13% auth bypass. |source:[independent-research]:https://owasp.org/www-project-mcp-top-10/:T1

R[26.4.5]: CyberArk extended attack surface finding — malicious instructions NOT limited to description field. Embedding also confirmed in: function names, parameter type labels, required[] arrays, default values, enum values. Implication: sanitization must scan ALL schema fields, not just .description. |source:[independent-research]:https://www.cyberark.com/resources/threat-research-blog/poison-everywhere-no-output-from-your-mcp-server-is-safe:T2

R[26.4.5]: Prompt injection in tool RESULTS — scope beyond description poisoning. Log-To-Leak paper (OpenReview 2025): MCP tool results (e.g., web search returning attacker-controlled pages, file contents) can contain injected instructions that model follows as if legitimate. Attack chain: tool result → injected instruction → model calls another tool → lateral movement or exfiltration. This is H[8] — CONFIRMED as documented attack vector. |source:[independent-research]:https://openreview.net/forum?id=UVgbFuXPaO:T1

R[26.4.5]: Prompt injection prevalence (2025): appeared in 73% of production AI deployments; only 34.7% of orgs deployed dedicated defenses. Agentic AI amplifies impact: single manipulated output can hijack agent planning, execute privileged tool calls, persist in memory, propagate across connected systems. |source:[independent-research]:https://christian-schneider.net/blog/prompt-injection-agentic-amplification/:T2

### Ollama tool calling state

R[26.4.5]: Ollama tool calling — native /api/chat endpoint. Format: tools array with type:"function" + function:{name,description,parameters} where parameters is JSON Schema (type:object, properties, required). Tool results fed back via messages with role:"tool". Format identical to OpenAI function calling spec — clean MCP→Ollama translation path exists. |source:[independent-research]:https://docs.ollama.com/capabilities/tool-calling:T1

R[26.4.5]: CRITICAL — streaming + tool calling incompatibility on /v1/chat/completions (OpenAI-compat endpoint). When stream:true + tools present, tool_calls delta chunks are silently dropped — response returns empty content with finish_reason:"stop". Workaround: stream:false on /v1 endpoint. BETTER solution: use native /api/chat endpoint, which has supported streaming + tool calling simultaneously since May 2025. Implication for bridge: MUST use /api/chat natively, NOT /v1 compat layer. |source:[independent-research]:https://github.com/ollama/ollama/issues/12557:T1 + https://www.betterclaw.io/blog/openclaw-ollama-guide:T2

### Security architecture patterns

R[26.4.5]: "Separate decide from do" — canonical architecture pattern for untrusted LLM output. Treat every LLM output token as tainted until validated. Policy layer OUTSIDE the LLM: assembles prompts, tags/strips untrusted content, enforces structured outputs, mediates every tool call. LLM = untrusted compute generating intent; policy layer = validates+translates intent into permitted actions. Directly applies to bridge design: LLM output → SecurityGateway → MCP call. |source:[independent-research]:https://arxiv.org/pdf/2506.08837:T1

R[26.4.5]: MiniScope paper (2025) — first formal least-privilege framework for tool-calling agents. Mechanical enforcement (not prompting): scopes each tool call's permissions to minimum required for that specific task. Principle: permissions expire after task completion. Applies to our bridge: per-session tool allowlists, not global grants. |source:[independent-research]:https://arxiv.org/pdf/2512.11147:T1

R[26.4.5]: Prompt Flow Integrity (PFI) paper (2025) — agent isolation + secure untrusted data processing + privilege escalation guardrails. Core insight: label data provenance (user-trusted vs tool-result-untrusted) and restrict what untrusted-labelled content can trigger. Applies to bridge: tool results should be wrapped/tagged before returning to model context to limit injection surface. |source:[independent-research]:https://arxiv.org/html/2503.15547v2:T1

R[26.4.5]: Goal-lock mechanisms — prevent model from abandoning original task via injected instructions. Implementation: system prompt includes explicit task description; bridge monitors whether tool calls remain consistent with original task scope; deviation triggers human checkpoint. |source:[independent-research]:https://sombrainc.com/blog/llm-security-risks-2026:T2

R[26.4.5]: Second-order prompt injection (2025 research) — low-privilege agent receives malformed request, tricks higher-privilege agent into unauthorized action. Applies to multi-agent chatroom scenario: compromised model in chatroom could inject into another model's context via shared memory (sigma-mem). Defense: treat inter-agent messages as untrusted input, same as external tool results. |source:[independent-research]:https://christian-schneider.net/blog/prompt-injection-agentic-amplification/:T2

R[26.4.5]: Ollama CVEs — bridge must NOT expose Ollama API to network. CVE-2025-63389: missing auth on Ollama API (≤0.12.3 but pattern persists). CVE-2025-51471: cross-domain token exposure via WWW-Authenticate header manipulation. CNVD-2025-04094: unauthenticated API access default config. SSRF exploited via model pull functionality (91,403 attack sessions Oct 2025–Jan 2026). Bridge must bind only to localhost, never expose Ollama port externally. |source:[independent-research]:https://github.com/advisories/GHSA-f6mr-38g8-39rg:T1 + https://nvd.nist.gov/vuln/detail/CVE-2025-51471:T1

### Tool sanitization approaches

R[26.4.5]: mcp-scan (Invariant Labs) — standard MCP security scanner. Detects: tool poisoning, rug pulls, cross-origin escalations, prompt injection in tool metadata. Scans: names, descriptions, schemas. Limitation: only analyzes metadata statically — does not catch dynamic/runtime generation of malicious outputs. Can be run as pre-flight check; incorporate similar pattern matching in bridge's tool ingestion layer. |source:[independent-research]:https://invariantlabs.ai/blog/mcp-security-notification-tool-poisoning-attacks:T1

R[26.4.5]: Sanitization scope (from CyberArk research) — ALL schema fields must be scanned. Detection targets: (a) embedded linguistic prompts in description/name/parameter-description, (b) suspicious length (benign tool descriptions rarely exceed 200 chars; >500 chars = high suspicion), (c) instruction-style language (imperative verbs: "ignore", "always", "never", "do not", "you must"), (d) cross-tool references ("when tool X runs", "override previous"), (e) exfiltration patterns (URLs, base64, webhook endpoints in descriptions), (f) privilege escalation language ("as administrator", "with full access"). |source:[independent-research]:https://www.cyberark.com/resources/threat-research-blog/poison-everywhere-no-output-from-your-mcp-server-is-safe:T2 + https://labs.snyk.io/resources/detect-tool-poisoning-mcp-server-security/:T2

R[26.4.5]: Hash-based rug pull defense — store SHA-256 of approved tool definition (name+description+parameters serialized) at approval time. On each session reconnect, recompute hash and compare. Any mismatch → block + alert user + require re-approval. Cost: near-zero at runtime. Implementation: simple dict {tool_id: hash} persisted to disk. |source:[independent-research]:https://mcpmanager.ai/blog/mcp-rug-pull-attacks/:T2

R[26.4.5]: Tool result sanitization — wrap all tool results before returning to model. Strip/escape: instruction-style text, role-prefix patterns ("SYSTEM:", "USER:", "ASSISTANT:"), markdown that could be parsed as prompts, excessive length (truncate + summarize). Tag provenance: prepend "[TOOL RESULT — UNTRUSTED]" to all tool outputs before injecting into conversation. This exploits model's own attention mechanism: labeled-untrusted content gets less instruction-following weight in practice. Empirical support from PFI paper. |source:[independent-research]:https://developer.microsoft.com/blog/protecting-against-indirect-injection-attacks-mcp:T2

### Threat model (STRIDE preliminary)

SPOOFING:
S1 — Malicious MCP server impersonating legitimate server (e.g., attacker-controlled "sigma-mem" clone). Mitigation: server allowlist by path/command, not name; hash verification of server binary on connect.
S2 — Tool result spoofing — injected content impersonates system/user turn. Mitigation: structured message wrapping, role enforcement in tool result injection.

TAMPERING:
T1 — Rug pull: tool definition modified post-approval. Mitigation: hash-based change detection, re-approval gate.
T2 — Parameter injection: model-generated args contain path traversal (../../etc/passwd), shell metacharacters (;, |, &&), SQL injection, template injection. Mitigation: parameter schema validation, type enforcement, path canonicalization, character allowlisting for string params.
T3 — Tool result injection: attacker-controlled content in tool results (web pages, files) modifies model behavior. Mitigation: result sanitization + provenance tagging.

REPUDIATION:
R1 — No record of what tools were called with what parameters. Mitigation: structured audit log (timestamp, tool, params, result_hash, model).
R2 — Model denies tool call was made (multi-agent scenario). Mitigation: cryptographic-style audit log with session ID.

INFORMATION DISCLOSURE:
I1 — Exfiltration via tool results: model extracts secrets (SSH keys, env vars, API keys) into tool call parameters sent to attacker-controlled server. Mitigation: server allowlist prevents calls to unregistered servers; output scanning for credential patterns.
I2 — Ollama API unauthenticated exposure (CVE-2025-63389 pattern). Mitigation: localhost-only binding, firewall rules.
I3 — Context over-sharing between sessions. Mitigation: session isolation, no shared state between model invocations unless explicit.

DENIAL OF SERVICE:
D1 — Infinite tool call loops (model keeps calling tools). Mitigation: max_turns limit, per-session tool call budget.
D2 — Large result payloads exhausting context window. Mitigation: result size limits (configurable per tool).
D3 — Rate-limit abuse via model-driven rapid calls. Mitigation: per-server rate limiting.

ELEVATION OF PRIVILEGE:
E1 — Model accesses MCP server not in approved list (lateral movement). Mitigation: server allowlist enforced by bridge, not model.
E2 — Tool calls outside approved tool list for session. Mitigation: tool allowlist per-session.
E3 — Destructive operations (write, delete, exec) without human confirmation. Mitigation: destructive action classification + confirmation gate.
E4 — Second-order injection via inter-agent messages (chatroom scenario). Mitigation: treat all inter-agent inputs as untrusted.

### Security requirements (preliminary)

SR-1 [CRITICAL]: Tool description sanitization at ingestion. Scan ALL schema fields (name, description, parameter names, parameter descriptions, defaults, enums) for: instruction-language patterns, length anomalies (>500 chars suspicious), cross-tool references, exfiltration patterns, privilege-escalation language. ¬proceed if BLOCK-level finding without explicit user override.

SR-2 [CRITICAL]: Hash-based rug pull detection. SHA-256 of serialized tool definition stored at approval. Recheck on reconnect. Mismatch → block + require user re-approval.

SR-3 [CRITICAL]: Server allowlist — whitelist by command path (not name). ¬connect to MCP servers not explicitly registered. Bridge enforces this — model cannot request new server connections.

SR-4 [CRITICAL]: Tool allowlist per session — even on approved servers, only tools explicitly permitted may be called. Default-deny, not default-allow.

SR-5 [CRITICAL]: Parameter validation — all model-generated parameters validated against tool's JSON Schema before MCP call. String params: length limit, character allowlist or denylist for dangerous metacharacters. Path params: canonicalize + check against permitted directories. ¬pass model-generated params directly to MCP without validation.

SR-6 [HIGH]: Tool result sanitization — all results wrapped with provenance tag before injecting to model context. Scan for role-prefix patterns, excessive instruction-style language. Truncate at configurable size limit.

SR-7 [HIGH]: Destructive action gate — classify tools as READ/WRITE/DESTRUCTIVE at registration. DESTRUCTIVE tools require human confirmation (Y/N prompt) before execution. Configurable: can downgrade to logging-only for automated pipelines.

SR-8 [HIGH]: Structured audit log — every tool call: timestamp, session_id, server, tool_name, params (sanitized — no secrets), result_size, result_hash, duration_ms, decision(allowed/blocked/confirmed). JSON-L format.

SR-9 [HIGH]: Rate limiting — per-server and per-tool call limits within session. Max-turns limit per agent invocation. Configurable via bridge config.

SR-10 [MEDIUM]: Ollama localhost binding — bridge communicates with Ollama via 127.0.0.1 only. ¬expose Ollama port. Bridge itself: configurable but default localhost.

SR-11 [MEDIUM]: Session isolation — no shared mutable state between concurrent sessions. Tool call loop state is per-session-scoped.

SR-12 [MEDIUM]: Result size limits — configurable max_result_bytes per tool. Oversized results: truncate + append "[TRUNCATED — result exceeded {n} bytes]".

SR-13 [LOW]: mcp-scan compatibility — bridge's tool ingestion scanner should detect the same patterns mcp-scan detects (tool poisoning, cross-origin escalation patterns) to allow offline/embedded scanning without external tool dependency.

### ADR[1]: Overall Library Architecture

**decision**: 5-layer architecture with SecurityGateway as mandatory pass-through

### ADR[3]: Tool Schema Translation Strategy

**decision**: Bidirectional translator module. MCP JSON Schema → Ollama tools format (for sending to model) + Ollama tool_call → MCP call_tool args (for executing).

**MCP tool schema** (from SDK list_tools):
```python
{
    "name": "store_memory",
    "description": "Store a memory entry",
    "inputSchema": {  # Full JSON Schema
        "type": "object",
        "properties": {
            "key": {"type": "string", "description": "Memory key"},
            "value": {"type": "string", "description": "Memory value"}
        },
        "required": ["key", "value"]
    }
}
```

**Ollama tools format** (for /api/chat):
```python
{
    "type": "function",
    "function": {
        "name": "store_memory",
        "description": "Store a memory entry",
        "parameters": {  # JSON Schema subset
            "type": "object",
            "properties": {
                "key": {"type": "string", "description": "Memory key"},
                "value": {"type": "string", "description": "Memory value"}
            },
            "required": ["key", "value"]
        }
    }
}
```

**translation is near-identity**: MCP inputSchema maps directly to Ollama parameters. Wrapper adds `type: "function"` + nests under `function` key. H[2] CONFIRMED — no lossy translation for standard schemas. |source:[workspace-research]:docs.ollama.com

**edge cases requiring handling**:
- Nested $ref in JSON Schema — resolve before passing to Ollama (small models may not handle $ref)
- anyOf/oneOf — flatten to most permissive type with description annotation
- Default values — preserve in schema, Ollama may or may not use them
- Enum values — preserve (but security gateway scans these per CyberArk research)

**namespace collision**: Multiple servers may have tools with same name. Solution: prefix with server name in Ollama tools array (`sigma-mem__store_memory`), strip prefix before MCP call_tool. Transparent to model via description. Consumer can configure delimiter.

**alternatives considered**:
- ALT-A: Pass-through with no translation — rejected: namespace collision unsolved, no $ref resolution
- ALT-B: Ollama Python library auto-schema from type hints — rejected: we receive MCP schemas at runtime, not at code time

|source:[workspace-research]+[independent-research]:docs.ollama.com |P=0.85

### ADR[4]: Agent Tool-Call Loop Design

**decision**: Multi-turn async loop with configurable limits, streaming support, and model-aware error recovery.

**loop pseudocode** (REVISED per DA[#6] — SecurityGateway.execute_tool() is atomic):
```
async def execute(prompt, model, max_turns=10):
    messages = [system_msg, user_msg(prompt)]
    tools = translator.get_ollama_tools()  # already translated+namespaced
    
    for turn in range(max_turns):
        response = await ollama.chat(model, messages, tools, stream=True)
        
        if no tool_calls in response:
            return FinalResult(response.content, audit)
        
        for tool_call in response.tool_calls:
            # SecurityGateway.execute_tool() is atomic: validate+call+sanitize
            # AgentLoop NEVER calls MCPClientManager directly
            try:
                result = await security.execute_tool(tool_call)
                # result is ExecutionResult: already validated, executed, sanitized
                messages.append(tool_result(tool_call.name, result.content))
            except ToolBlockedError as e:
                messages.append(tool_result(tool_call.name, f"BLOCKED: {e.reason}"))
            except ConfirmationDeniedError:
                messages.append(tool_result(tool_call.name, "DENIED by user"))
            except ToolError as e:
                messages.append(tool_result(tool_call.name, f"ERROR: {e.safe_message}"))
    
    return FinalResult("Max turns reached", audit, truncated=True)
```

**streaming**: Use native /api/chat (CRITICAL — /v1/chat/completions silently drops tool_calls when streaming) |source:[workspace-research]:github.com/ollama/ollama/issues/12557. Stream text tokens to callback as they arrive. Buffer tool_call tokens until complete, then process.

**error recovery** (model-aware, addresses H[1] challenge):
- Malformed tool_call (invalid JSON in args): retry with correction prompt ("Your tool call had invalid JSON. The correct format is: {schema}"). Max 2 retries per call.
- Unknown tool name: inform model, list available tools. (Models hallucinate tool names.)
- Tool execution error: return error as tool result, let model decide next action.
- Model returns empty content + no tool_calls: treat as "model is stuck", inject nudge message. Max 1 nudge per turn.
- gemma4:e4b specific: if tool_calls present but unparseable (known bug #15315), attempt regex extraction of function name + args from raw text. Fallback only. |source:[workspace-research]:github.com/ollama/ollama/issues/15315

**max_turns**: default 10, configurable. Hard cap at 50 (prevent runaway). Per-session tool call budget also enforced (SR-9).

**alternatives considered**:
- ALT-A: Single-shot (no loop) — rejected: multi-tool workflows require multi-turn
- ALT-B: Plan-then-execute (model outputs full plan, bridge executes all at once) — rejected: models at 4B-8B can't reliably plan multi-step. Iterative loop is more robust.
- ALT-C: Parallel tool execution (call multiple tools concurrently) — deferred to v1.1. V1 executes sequentially for simpler security auditing. Can add concurrent execution later.

|source:[workspace-research]:H[1]+H[5]+C[4] |P=0.65 (model-specific bugs create uncertainty)

### ADR[5]: Multi-Server Management

**decision**: MCPClientManager maintains concurrent connections to multiple MCP servers. Lazy connect on first tool call to that server. Graceful lifecycle management.

**design**:
```python
class MCPClientManager:
    servers: dict[str, ServerConnection]  # name → connection
    
    async def connect(self, name: str, config: ServerConfig) -> None
    async def disconnect(self, name: str) -> None
    async def disconnect_all(self) -> None
    async def list_tools(self, name: str) -> list[ToolSchema]
    async def list_all_tools(self) -> dict[str, list[ToolSchema]]
    async def call_tool(self, server: str, tool: str, args: dict) -> ToolResult
    async def health_check(self, name: str) -> ServerHealth
```

**lifecycle**:
- Servers configured at Bridge init (from config)
- Connections established lazily (first tool call or explicit connect)
- stdio processes managed as async subprocesses
- Disconnect: graceful shutdown signal → timeout → force kill
- Bridge.__aexit__ disconnects all servers
- Reconnect on transient failure (max 3 attempts, exponential backoff)

**server identity**: by config path/command (SR-3), not name. Two servers cannot share the same command+args. Prevents server spoofing (S1 from STRIDE).

**tool namespace**: `{server_name}__{tool_name}` separator. Server name from config (user-chosen, validated alphanumeric+hyphen). Tool names from MCP list_tools. Combined namespace presented to Ollama model.

**resource limits**:
- Max 10 concurrent server connections (configurable)
- Per-server: max tool calls/minute (SR-9), max result size (SR-12)
- Total: max tool calls across all servers per session

**alternatives considered**:
- ALT-A: Single server only — rejected: Q[4] requires multi-server (sigma-mem + sigma-verify + file ops + web search)
- ALT-B: Sequential connections (connect/use/disconnect per call) — rejected: MCP init handshake per call adds 100-500ms latency
- ALT-C: Connection pool with pre-warming — deferred: lazy connect is simpler, pre-warming adds complexity for marginal benefit at <10 servers

|source:[workspace-research]:Q[4]+SR-3 |P=0.80

### ADR[8]: Error Handling and Graceful Degradation

**decision**: Exception hierarchy + fallback chain + never-crash contract.

**exception hierarchy**:
```python
class BridgeError(Exception): pass

# Transport errors
class OllamaError(BridgeError): pass
class OllamaConnectionError(OllamaError): pass  # can't reach Ollama
class OllamaModelError(OllamaError): pass        # model not found/loaded
class OllamaResponseError(OllamaError): pass     # malformed response

class MCPError(BridgeError): pass
class MCPConnectionError(MCPError): pass   # server won't start
class MCPToolError(MCPError): pass         # tool execution failed
class MCPTimeoutError(MCPError): pass      # tool call timed out

# Security errors
class SecurityError(BridgeError): pass
class ToolBlockedError(SecurityError): pass      # tool not in allowlist
class ParameterRejectedError(SecurityError): pass  # param validation failed
class RateLimitError(SecurityError): pass          # rate limit exceeded
class ConfirmationDeniedError(SecurityError): pass # user denied destructive action
class ToolIntegrityError(SecurityError): pass      # hash mismatch (rug pull)

# Loop errors
class LoopError(BridgeError): pass
class MaxTurnsError(LoopError): pass     # hit turn limit
class StuckModelError(LoopError): pass   # model not making progress
```

**graceful degradation strategy** (C[4]):
1. **Model malforms tool call** → retry with correction prompt (max 2), then skip tool + inform model
2. **MCP server unreachable** → disable that server's tools for session, inform model, continue with remaining servers
3. **Tool execution fails** → return error as tool result, model decides next action
4. **Ollama unreachable** → raise OllamaConnectionError to consumer (can't degrade — Ollama IS the model)
5. **Security blocks a call** → inform model "tool call blocked: {reason}", model can try different approach
6. **Rate limit hit** → backoff + inform model "rate limit reached, waiting {n}s", auto-retry
7. **Max turns reached** → return partial result with truncated=True, include what was accomplished

**never-crash contract**: Bridge.run() NEVER raises from tool execution failures or model misbehavior. It returns BridgeResult with error information. Only raises BridgeError subclasses for infrastructure failures (can't reach Ollama, config invalid, etc.).

**alternatives considered**:
- ALT-A: Error codes instead of exceptions — rejected: Python idiom is exceptions, error codes lose stack traces
- ALT-B: Result monad (Ok/Err) — rejected: non-standard in Python ecosystem, learning curve for consumers

### ADR[9]: HATEOAS-Driven Dynamic Tool Presentation (hateoas-agent integration)

**decision**: Integrate hateoas-agent as an OPTIONAL Layer 2.5 between SecurityGateway and ToolTranslator for state-driven tool filtering. When enabled, the bridge presents only contextually relevant tools to the model per turn, rather than the full tool list.

**problem this solves**:
- Small models (4B-8B) degrade rapidly with large tool lists. Qwen 3.5 switches to XML-in-content at >5 tools |source:[workspace-research]:github.com/ollama/ollama/issues/14493
- Tool hallucination (PM[4], P=0.35) scales with tool count — fewer visible tools = fewer hallucination targets
- Security surface: presenting 30 tools when only 3 are relevant exposes 27 unnecessary attack vectors per turn
- README comparison table quantifies: flat tools = "100+ visible per turn", hateoas-agent = "3-5 (only what's valid)" |source:[codebase-inspection]:~/Projects/hateoas-agent/README.md

**how hateoas-agent works** (from codebase review):
- `Registry` wraps a `HasHateoas` resource (StateMachine or Resource)
- `Registry.get_current_tool_schemas()` returns ONLY gateway + actions valid in current state
- State transitions tracked via `_last_state` — each handler return includes `_state` key
- `CompositeRegistry` merges multiple resources (multi-server maps to multi-resource)
- `filter_actions()` applies guard functions for additional dynamic filtering
- Already used in sigma-mem: `StateMachine("sigma_mem", gateway_name="recall")` with state-dependent action advertisement |source:[codebase-inspection]:~/Projects/sigma-mem/src/sigma_mem/machine.py

**integration design**:
```
                          ┌─────────────────────────────┐
                          │ Layer 2.5: ToolPresenter     │
                          │  (hateoas-agent Registry)    │
                          │                              │
                          │  Mode A: HATEOAS (default    │
                          │    for ≤8B models)           │
                          │    → StateMachine per server │
                          │    → gateway: "use_{server}" │
                          │    → actions: server's tools │
                          │    → state: tracks last tool │
                          │      result context          │
                          │                              │
                          │  Mode B: FLAT (default for   │
                          │    >8B models)               │
                          │    → all approved tools      │
                          │    → no state tracking       │
                          │    → current ADR[4] behavior │
                          └─────────────────────────────┘
```

**HATEOAS mode specifics**:
- Each MCP server becomes a `StateMachine` resource
- Gateway tool per server: `use_sigma_mem`, `use_files`, etc. — model calls gateway to "enter" that server's context
- Once in a server's context, only that server's approved tools are visible (+ all gateways for switching)
- State transitions: `idle` → `{server}_active` → tool call → result triggers state with new available actions
- `CompositeRegistry` composes all server state machines — exactly the pattern it was built for
- Tool count per turn: N_gateways + tools_for_active_server. With 4 servers × 5 tools each = 4 gateways visible at start, then ~5+4=9 tools when in a server context. vs FLAT mode: 20 tools always visible.

**why OPTIONAL/modal**:
- Large models (70B+) handle 20+ tools fine — HATEOAS overhead adds unnecessary indirection
- Some consumers may want direct flat access (sigma-review cross-model verification = single tool call, no workflow)
- Backward-compatible: FLAT mode = current ADR[4] loop, no hateoas-agent dependency

**dependency implications**:
- hateoas-agent is already a dependency of sigma-mem (same ecosystem)
- Zero runtime deps in hateoas-agent itself (pure Python)
- Import is conditional: `if config.tool_presentation == "hateoas": from hateoas_agent import StateMachine, CompositeRegistry`
- Can be a pip extra: `pip install ollama-mcp-bridge[hateoas]`

**effect on other ADRs**:
- ADR[1]: adds optional Layer 2.5 module (`presenter.py`) — does NOT change the 5-layer mandatory security path
- ADR[3]: ToolTranslator still handles MCP↔Ollama format conversion. ToolPresenter handles WHICH tools, ToolTranslator handles HOW they're formatted.
- ADR[4]: AgentLoop.execute() calls `presenter.get_current_tools()` instead of `translator.get_all_tools()`. Security gateway still validates every call regardless of presentation mode.
- ADR[5]: MCPClientManager unchanged. ToolPresenter is built FROM the approved tool list, not from raw MCP schemas.
- PM[4]: revised from P=0.35 → P=0.15 in HATEOAS mode (3-5 tools vs 20+)

**alternatives considered**:
- ALT-A: Tool RAG (embed tool descriptions, retrieve semantically) — rejected: probabilistic retrieval ¬deterministic, adds embedding dependency, hateoas-agent README comparison table shows RAG is "probabilistic" vs HATEOAS "deterministic"
- ALT-B: LLM-based tool search (model writes query to find tools) — rejected: adds extra LLM call per turn, latency, still probabilistic
- ALT-C: Always HATEOAS, no flat mode — rejected: overhead for large models, some use cases need direct access
- ALT-D: Build custom state-based filtering from scratch — rejected: hateoas-agent already implements this exact pattern with tests, sigma-mem already validates it in production use

|source:[codebase-inspection]:~/Projects/hateoas-agent/ + [workspace-research]:PM[4]+Qwen-bug |P=0.75

## security-architecture (output of r1-r2 — locked after DA approval)

### SAD[1]: Permission Model — Default-Deny Per-Session Allowlists
|status:DESIGNED |priority:CRITICAL |implements:SR-3,SR-4 |mitigates:E1,E2
|XVERIFY[openai:gpt-5.4]:AGREE,high-confidence |source:[independent-research]+[external-verification]

**decision**: default-deny with per-session tool allowlists. No server connects, no tool executes without explicit PermissionPolicy entry.

**types**:
- `PermissionPolicy`: servers(list[ServerPermission]), tools(list[ToolPermission]), max_turns(int), max_calls_per_tool(dict[str,int])
- `ServerPermission`: command_path(str—absolute), args(list[str]—frozen), allowed_tools(list[str]|None), trust_level(TRUSTED|STANDARD|SANDBOXED)
- `ToolPermission`: server_id, tool_name, action_class(READ|WRITE|DESTRUCTIVE), requires_confirmation(bool), max_calls(int|None)

**behavior**: session-scoped+immutable | escalation→"Permission denied" to model ¬silent drop | named config profiles | JSON serialized for audit
**enforcement**: `SecurityGateway.authorize()` BEFORE every MCP call ¬advisory
**alternatives rejected**: default-allow+denylist(brittle), RBAC(single trust level, future extensibility per XVERIFY)

### SAD[2]: Tool Description Sanitization Pipeline
|status:DESIGNED |priority:CRITICAL |implements:SR-1,SR-13 |mitigates:S1,E1,E2,E3
|source:[independent-research]:CyberArk+InvariantLabs+Snyk:T1/T2

**decision**: 3-stage pipeline at tool ingestion
- **EXTRACT**: ALL text fields ¬description-only (CyberArk: poison in ANY schema field)
- **SCORE**: 7 detectors — instruction_lang(80-100), cross_tool_ref(60-80), exfil_pattern(70-90), priv_escalation(70-90), length_anomaly(additive), role_impersonation(90-100), encoding_obfuscation(80-100). Aggregate=max(field_scores)
- **DECIDE**: PASS(<40), WARN(40-69: sanitize substrings), BLOCK(≥70: user override available)
- **impl**: `ToolSanitizer` + pluggable `SanitizationRule` protocol

### SAD[3]: Parameter Validation Architecture
|status:DESIGNED |priority:CRITICAL |implements:SR-5 |mitigates:T2
|source:[independent-research]:OWASP-MCP-Top10:T1

**decision**: 3-layer validation before every MCP call
- **L1 Schema**: `jsonschema`, strict additionalProperties=false, zero coercion
- **L2 Type-Specific**: String(length+metachar), Path(realpath+allowed_dirs), URL(scheme allowlist), Number(range+¬NaN), Array(max_items), NestedObject(max_depth=5→GAP-6)
- **L3 Contextual**: per-tool in PermissionPolicy
- **enforcement**: Valid|Invalid(field errors), max 3 retries then skip+log

### SAD[4]: Tool Result Sanitization Pipeline
|status:DESIGNED |priority:HIGH |implements:SR-6 |mitigates:T3,S2
|source:[independent-research]:PFI-paper:T1+Microsoft-MCP-defense:T2

**decision**: 3-stage — SIZE GATE(50KB default, truncate), CONTENT SCAN(role-prefix strip, instruction tag, URL flag), PROVENANCE WRAP("[UNTRUSTED EXTERNAL DATA]"). GAP-5: model-dependent, defense-in-depth.

### SAD[5]: Audit Logging Architecture
|status:DESIGNED |priority:HIGH |implements:SR-8 |mitigates:R1,R2

**decision**: JSON-L append-only. Schema: timestamp,session_id,event_type,server_id,tool_name,action_class,params_hash(¬raw),params_summary(200ch),result_size,result_hash,decision,reason,score,duration_ms,model_id,turn. Storage: `~/.ollama-mcp-bridge/audit/{date}.jsonl`, 30-day rotation. Async+buffered+thread-safe.

### SAD[6]: Action Gate Design
|status:DESIGNED |priority:HIGH |implements:SR-7 |mitigates:E3

**decision**: classify(explicit→name heuristic→default=WRITE) + gate(READ:auto, WRITE:auto(configurable), DESTRUCTIVE:human confirm Y/N/always-per-tool). `confirmation_mode="log-only"` explicit for automation. Pluggable callback, 60s timeout→Denied.

### SAD[7]: Hash-Based Rug Pull Detection
|status:DESIGNED |priority:CRITICAL |implements:SR-2 |mitigates:T1

**decision**: SHA-256 canonical tool def. Registry: `~/.ollama-mcp-bridge/approved_tools.json`. Check: every connect + every call. Mismatch→BLOCK+notify+re-approve triggers re-sanitize. New→sanitize first→prompt→registry.

### SAD[8]: Rate Limiting
|status:DESIGNED |priority:HIGH |implements:SR-9 |mitigates:D1,D2,D3

**decision**: 5 levels — max_turns(50), per_tool(20), per_server(100), temporal(30/min), result_budget(10MB). All configurable.

### SAD[9]: Session Isolation
|status:DESIGNED |priority:MEDIUM |implements:SR-11 |mitigates:I3

**decision**: no shared mutable state. Own MCP connections(fresh), conversation, rate limiter, gate state per session. 4h idle timeout.

### SAD[10]: Trust Boundaries
|status:DESIGNED |priority:CRITICAL |implements:SR-10,C[7] |mitigates:all-STRIDE
