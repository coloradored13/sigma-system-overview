#### IC[B3-2]: sigma_ui/execution_loop.py [REVISED]

dispatch_phase(
    phase: str,
    dispatcher: AgentDispatcher,
    context_builder: ContextBuilder,
    review_state: ReviewState,
    review_id: str,
    workspace_path: Path | None,
    phase_dispatch_map: dict[str, list[str]],
    round_num: int = 1,
) → list[AgentResult]

Dispatches all agents for a phase and records results to ReviewState. Blocking sync call.

Steps:
  1. agent_names = phase_dispatch_map.get(phase, [])
  2. For each agent_name: review_state.record_agent_start(review_id, agent_name, round_num, task=f"{phase} R{round_num}")
  3. Build AgentContext per agent via context_builder.build_pre_spawn_context()
  4. Build AgentSlot list: [AgentSlot(name=n, role="") for n in agent_names]
  5. Dispatch in ThreadPoolExecutor(max_workers=5):
       futures = {pool.submit(_run_one, slot, ctx): slot.name for slot, ctx in zip(slots, contexts)}
       where _run_one(slot, ctx): return asyncio.run(dispatcher.dispatch(slot, task_str, ctx))
  6. For each result: parse_findings → add_finding; asyncio.run(record_convergence) in thread
  7. context_builder.persist_post_response per agent (non-fatal)
  8. review_state.compute_and_store_belief(review_id, workspace_content, round_num)
  9. Return list[AgentResult]

_build_phase_dispatch_map() → dict[str, list[str]]:
  Reads orchestrator-config.py _default_analyze_agents() agent names.
  Returns: {"research": [names_except_da], "challenge": [all_names_plus_da], "circuit_breaker": [], "synthesis": [], ...}
  Empty list for phases with no agents = sigma-ui skips dispatch, proceeds to gate/advance.

BLOCKING NOTE: dispatch_phase() is sync and blocks until all agents complete.
  For trivial E2E test (1 agent): ~5-15s — acceptable with st.spinner.
  For full 20-agent review: ~60-180s blocking — Phase B4 concern (background threading). Out of B3 scope per C[2].

parse_findings_from_response(text) → list[dict]:
  Extract F[N] markers OR fallback to response body as single finding.
  Each: body, source_tag="agent-inference", source_tier="T3"

#### DS[B3-2]: Dismiss-as-Cancel Pattern

MECHANISM: G2 has dismissible=True (default). When user clicks outside:
  1. Streamlit closes dialog, triggers rerun
  2. app.py rerun: active_gate still non-None (no session state change from dismiss)
  3. tier = route_gate_tier(active_gate) → "TIER-2"
  4. render_user_gate() called again → @st.dialog() called again → dialog re-opens
  5. Net effect: dismiss = dialog temporarily closes, immediately re-opens on same rerun → appears as "no-op" to user

SESSION STATE CHANGE on dismiss: NONE. active_gate unchanged. gate_passed=False. gate_outcome=None.
USER EXPERIENCE: dismiss is equivalent to "nothing happened" — dialog reappears. This is correct behavior (conservative default — gate stays open).

DISMISS vs CANCEL semantic distinction:
  DISMISS (outside-click): no session state change, dialog loops (acceptable UX, not ideal)
  CANCEL (explicit button): ss["gate_passed"]=False, ss["gate_outcome"]=None — same as dismiss but communicates user intent explicitly. No advance() call triggered. Gate stays open. Dashboard visible.

WHY NOT set dismiss to "EXTEND"? Because dismiss is uncatchable in Streamlit — no callback fires on outside-click. The only mechanism is: dialog re-opens because gate persists. Explicit Cancel button is the authoritative action. Dismiss is a fallback.

POST-CANCEL STATE: gate stays open (active_gate non-None). Dialog re-opens on next rerun. User must explicitly Advance or Extend to resolve gate. No way to "dismiss forever" — this is correct for gate semantics.

GATE PERSISTENCE AFTER CANCEL: active_gate cleared only by resolve_gate() called via _advance_and_clear(). Cancel = no advance() call = no resolve_gate() = gate persists. This is the correct invariant.
