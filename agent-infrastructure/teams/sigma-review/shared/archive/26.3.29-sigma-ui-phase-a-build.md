- Async: sigma-verify sync clients = design defect in async context. Thin adapter class required.
- Memory lifecycle: pre-spawn read (handle_get_agent_memory) + post-response write (handle_store_agent_memory) = atomic unit. Ship both or neither.

- RISK-1 (HIGH): Guard gaming residual — agents write well-formed compliance artifacts without genuine work. Mitigation: TIER-A observables.

- RISK-3 (HIGH): AsyncRunner private-attr bypass. Mitigation: don't use AsyncRunner.
- RISK-4 (HIGH): Blocking-call event-loop stall. Mitigation: async adapter class.

H3 is the load-bearing uncertainty: P1 context injection may degrade analytical quality. Review rated this at P=52-62%. This hypothesis MUST be genuinely challenged in plan phase — if evidence shows P1 insufficient, architecture changes (P2 moves into Phase A, or dispatch model shifts). The plan adapts to findings, not the other way around.

### SC[] — Stagger contract (H2/H6, OQ-3 resolution)

#### SC1: Responsibility assignment
Convergence-write stagger (>=500ms between workspace state updates during phase transitions) is an **ORCHESTRATOR responsibility**, NOT a data layer responsibility.

Rationale:
- Data layer (ReviewSnapshot) is read-only from Phase B perspective — it cannot enforce write timing
- The stagger prevents Streamlit rerun storm (PD memory C[Streamlit-viability: burst-rate-¬average-rate-is-binding-constraint])
- Only the orchestrator controls WHEN it writes convergence declarations to workspace
- A data-layer rate-limiter would buffer writes invisibly, breaking the audit trail

#### SC2: Write-path stagger contract (REVISED per DA[#6]+BC[IE-8] concede)
Two distinct staggers — different purposes, different locations:

**Stagger A — dispatch-start stagger (TA IC[TA-2] dispatch_parallel stagger_ms):**
Purpose: API rate-limit protection + load distribution. Located in AgentDispatcher.dispatch_parallel().
This does NOT satisfy the SC2 write-path contract. Agents that start 500ms apart can still complete near-simultaneously.

**Stagger B — write-path stagger (SC2 contract — Phase B relies on this):**
Purpose: prevent Streamlit rerun storm from burst convergence events.
Location: whichever module accumulates AgentResult objects after dispatch completes.
- If ReviewState is Phase A scope (BC[IE-7] option a): stagger lives in ReviewState.record_result()
- If ReviewState is Phase B scope (BC[IE-7] option b, recommended): stagger lives in Phase B's ReviewState
Phase A requirement (regardless of scope decision): asyncio.Semaphore(1) on any workspace write path that multiple agents share. This prevents concurrent workspace writes corrupting state even without the stagger timing.

```python
# Phase A MUST have: asyncio.Semaphore(1) on shared workspace write paths
_write_lock = asyncio.Semaphore(1)

async def write_agent_output(agent_id: str, output: str):
    async with _write_lock:
        # serialize write — only one agent writes at a time
        await write_workspace_section(agent_id, output)
        # stagger B: sleep AFTER write, before lock release
        await asyncio.sleep(CONVERGENCE_WRITE_STAGGER_MS / 1000)
```
!The sleep is INSIDE the semaphore context — this ensures minimum inter-write spacing, not just mutual exclusion.
!dispatch_parallel stagger_ms (Stagger A) is a SEPARATE parameter on AgentDispatcher — do not remove it, but document its purpose as rate-limit protection, not write-path serialization.
!CONVERGENCE_WRITE_STAGGER_MS default: 500ms. Configurable. Phase B must poll at >= stagger_ms × 2 interval.

→ OQ-4: stagger write-path owner depends on BC[IE-7] ReviewState scope decision. Lead must resolve before build-track starts.

#### SC3: Phase B contract (what Phase B relies on)
- Phase B polling interval MUST be >= stagger_ms (default: poll every max(2000ms, stagger_ms×2))
- Phase B must NOT assume atomic multi-agent convergence — agents converge one at a time with stagger
- Phase B fragment key=agent_id isolates per-card reruns; ReviewSnapshot.last_write_at change triggers only affected card rerun

#### SC4: Stagger-infeasibility escalation (OQ-3 resolution path)
If Phase A build-track determines orchestrator cannot guarantee stagger (e.g., external agents converge faster than stagger can enforce), this must be flagged as OQ-3 resolution: INFEASIBLE.
