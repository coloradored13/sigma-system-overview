# Synthesis: Ollama MCP Bridge -- sigma-build
## Date: 2026-04-05

## Prompt Decomposition

**Questions:**
- Q[1]: Python library connecting any Ollama model to any MCP server
- Q[2]: MCP tool schema to Ollama function calling format translator
- Q[3]: Multi-turn tool call loop (model calls tool, bridge executes, feeds result, model continues)
- Q[4]: Multi-server support (sigma-mem, sigma-verify, file ops, web search simultaneously)
- Q[5]: Standalone, importable library consumed by chatroom, sigma-review, or anything else

**Hypotheses:**
- H[1]: Ollama's function calling API is mature enough for reliable multi-tool use
- H[2]: MCP tool schemas (JSON Schema) can be cleanly mapped to Ollama's function format without lossy translation
- H[3]: stdio transport is the right MCP transport choice (vs SSE) for local-to-local communication
- H[4]: A Python MCP client library already exists (the `mcp` package) -- should we use it or build lightweight?
- H[5]: The tool call loop can be generalized (not model-specific) since Ollama normalizes function calling across model families
- H[6]: Local models have zero built-in safety training for tool use -- the bridge is the ONLY security layer
- H[7]: Tool description poisoning (hidden instructions in MCP tool metadata) is the primary attack vector
- H[8]: Tool results can contain prompt injection -- a web search result or file contents could contain instructions the model follows

**Constraints:**
- C[1]: Python -- matches existing infrastructure (sigma-mem, sigma-verify, Ollama API)
- C[2]: Model-agnostic and hardware-agnostic -- works with 4B today, 70B+ tomorrow
- C[3]: Standalone library with minimal dependencies
- C[4]: Must handle graceful degradation (model malforms a tool call, retry/skip, don't crash)
- C[5]: Must work with current setup: Ollama 0.20.2 Pro, macOS, existing MCP servers
- C[6]: Security by design -- permission model, tool allowlisting, audit logging, human confirmation for destructive actions
- C[7]: Bridge must enforce security even if the model actively tries to circumvent it (compromised model scenario)
- C[8]: Defense in depth -- no single security mechanism is sufficient

## Hypothesis Outcomes

**H[1]: CHALLENGED (CONDITIONAL)**
Ollama function calling maturity is conditional. llama3.1:8b appears reliable. gemma4:e4b has known active bugs in Ollama 0.20.x (issue #15315). Bridge must implement retry + malformed-call recovery because model-level tool call formatting is NOT guaranteed even on supported models. ADR[4] confidence revised from P=0.65 to ~P=0.80 after adding SQ[0] empirical testing phase.

**H[2]: CONFIRMED**
Clean mapping exists. JSON Schema from MCP maps to Ollama's tools format (type:"function" + function:{name,description,parameters}). Translator handles namespace prefixing (server__tool), $ref resolution, anyOf flattening. Source: docs.ollama.com/capabilities/tool-calling (T1).

**H[3]: CONFIRMED**
stdio is correct transport for local-to-local. SSE is deprecated by MCP project. Streamable HTTP is for remote deployments. Source: pypi.org/project/mcp (T1).

**H[4]: CONFIRMED**
Official `mcp` Python package exists and is production-ready. Build decision: USE the official SDK for MCP protocol layer; build custom logic for security gateway, tool translation, and loop orchestration. Source: github.com/modelcontextprotocol/python-sdk (T1).

**H[5]: PARTIAL**
Ollama normalizes format across model families, BUT model-specific bugs exist (gemma4:e4b parsing errors, Qwen 3.5 agentic bugs, Qwen3-coder >5 tools switches to XML). Per-model error handling needed. Generic loop with model-specific error recovery.

**H[6]: CONFIRMED**
Extensive security research validates this. 30+ MCP CVEs Jan-Feb 2026. ZERO existing Ollama-MCP bridges implement ANY security features. Tool poisoning PoC confirmed (Invariant Labs: SSH key exfiltration via hidden instructions in tool descriptions). Bridge is the ONLY security layer.

**H[7]: CONFIRMED**
Tool description poisoning is a documented attack vector (CVE-2025-6514). CyberArk extended finding: malicious instructions embedded not just in description but in ALL schema fields (function names, parameter types, required arrays, default values, enum values). Implication: sanitization must scan ALL schema fields.

**H[8]: CONFIRMED**
Prompt injection in tool results is a documented attack vector. Log-To-Leak paper (OpenReview 2025): MCP tool results containing injected instructions that model follows. Attack chain: tool result -> injected instruction -> model calls another tool -> lateral movement or exfiltration.

## Architecture Decisions (ADR Summary)

**ADR[1]: 5-Layer Library Architecture with Security Gateway**
- Decision: Library (not proxy/server), 5 mandatory layers: L1(transport: Ollama+MCP), L2(translation), L3(security gateway), L4(agent loop), L5(consumer API)
- Rejected: monolithic, plugin/middleware, proxy
- Confidence: P=0.80
- Source: arxiv:2506.08837 "separate decide from do" (T1) + security-research aggregate (T2)
- XVERIFY[openai:gpt-5.4]: AGREE, high-confidence

**ADR[2]: Official MCP SDK + stdio Transport**
- Decision: Use official `mcp` Python SDK, stdio transport for local-to-local
- Rejected: custom lightweight client, SSE transport
- Confidence: P=0.90
- Source: github.com/modelcontextprotocol/python-sdk (T1), pypi.org/project/mcp (T1)

**ADR[3]: Bidirectional Schema Translation with Namespacing**
- Decision: MCP JSON Schema to Ollama tools format. Namespace: `{server_name}__{tool_name}`. Handle $ref resolution, anyOf flattening.
- Rejected: no namespace (collision risk), dot separator (conflicts with Python)
- Confidence: P=0.85
- Source: docs.ollama.com/capabilities/tool-calling (T1)

**ADR[4]: Multi-Turn Agent Loop with Error Recovery**
- Decision: While-loop pattern: model response -> extract tool_calls -> SecurityGateway.execute_tool() (atomic) -> feed result back -> continue. Streaming via native /api/chat (CRITICAL -- /v1/chat/completions silently drops tool_calls when streaming). Model-aware error recovery (malformed JSON retry, unknown tool inform, stuck model nudge, gemma4:e4b regex fallback).
- Rejected: single-shot, plan-then-execute, parallel tool execution (deferred to v1.1)
- Confidence: P=0.65 -> ~P=0.80 after SQ[0] empirical testing
- Source: github.com/ollama/ollama/issues/15315,12557 (T1)

**ADR[5]: Multi-Server Management with Lazy Connect**
- Decision: MCPClientManager maintains concurrent connections. Lazy connect on first tool call. Per-server AsyncExitStack for lifecycle. Server identity by config path/command (not name). Max 10 concurrent servers.
- Rejected: single server only, sequential connect/disconnect per call
- Confidence: P=0.80
- Source: Q[4] requirements (T2) + MCP SDK stdio transport (T1)

**ADR[6]: Single Bridge Class as Public API**
- Decision: Bridge class as sole public interface. Async-first with sync wrapper. Context manager for lifecycle. from_config() for TOML, programmatic config also supported. run/run_sync/run_stream methods.
- Rejected: functional API, builder pattern, multiple entry points
- Confidence: P=0.75
- Source: Q[5]+C[3] requirements (T2) + chatroom/sigma-review API needs (T3)

**ADR[7]: TOML Configuration with Pydantic Validation**
- Decision: TOML primary format, Pydantic BaseModel validation, programmatic override
- Rejected: YAML (indentation footgun, security history), JSON (no comments), Python file (code execution risk)
- Confidence: P=0.90
- Source: Python stdlib tomllib (T1)

**ADR[8]: Exception Hierarchy + Graceful Degradation**
- Decision: BridgeError hierarchy (Ollama/MCP/Security/Loop errors). 7 degradation scenarios + post-sanitization failure tiers (CQ-4: annotate/redact/quarantine). Never-crash contract: Bridge.run() never raises from tool execution failures.
- Rejected: error codes, Result monad
- Confidence: P=0.70
- Source: C[4]+H[1]+GAP-1 (T2), Ollama issue tracker (T1)

**ADR[9]: HATEOAS-Driven Dynamic Tool Presentation -- DEFERRED TO v1.1**
- Decision: Design preserved. When enabled, uses hateoas-agent StateMachine per MCP server to present only contextually relevant tools per turn (3-5 vs 20+). Reduces tool hallucination from P=0.35 to P=0.15.
- Deferred per user decision. v1 uses flat tool presentation. ToolPresenterProtocol interface allows v1.1 addition with zero API breakage.
- Confidence: P=0.75

**ADR[10]: Build-vs-Wrap -- Reuse ollama + mcp SDK Packages**
- Decision: Use `ollama` Python package (v0.6.1) as OllamaClient AND `mcp` SDK as MCPClient. SQ[3] and SQ[5] become thin wrappers, not reimplementations. jonigl/ollama-mcp-bridge inspected as reference for loop structure and MCP connection patterns.
- What we take: _connect_server() pattern, call_tool() SDK usage, loop structure (~40% reference)
- What we do NOT take: FastAPI proxy arch, zero security, flat tool presentation, raw error handling
- Effort savings: -1.75d (10.5d -> 8.75d)
- Confidence: P=0.85
- Source: codebase inspection of jonigl/ollama-mcp-bridge 0.11.2 + ollama 0.6.1 (T1)

## Security Architecture (SAD Summary)

**SAD[1]: Default-Deny Per-Session Allowlists** | CRITICAL | mitigates E1,E2
- Per-session immutable PermissionPolicy. No server connects, no tool executes without explicit entry. Session-scoped. XVERIFY[openai:gpt-5.4]: AGREE, high-confidence.

**SAD[2]: Tool Description Sanitization Pipeline** | CRITICAL | mitigates S1,E1,E2,E3
- 3-stage pipeline: EXTRACT all text fields (not description only, per CyberArk) -> SCORE via 7 detectors (instruction_lang, cross_tool_ref, exfil_pattern, priv_escalation, length_anomaly, role_impersonation, encoding_obfuscation) -> DECIDE (PASS <40, WARN 40-69, BLOCK >=70). NFKC normalization. Pluggable SanitizationRule protocol. 7 detectors retained per user mandate.

**SAD[3]: Parameter Validation -- 3 Layers** | CRITICAL | mitigates T2
- L1: JSON Schema (strict, additionalProperties=false, zero coercion). L2: Type-specific (string length+metachar, path canonicalization, URL scheme allowlist, NaN/Infinity rejection, array max_items, nested depth max=5). L3: Contextual (extensibility point, deferred concrete rules).

**SAD[4]: Tool Result Sanitization -- 4 Tiers** | HIGH | mitigates T3,S2
- SIZE GATE (50KB default, truncate) -> CONTENT SCAN (role-prefix strip, instruction tag detect, URL flag) -> PROVENANCE WRAP ("[UNTRUSTED EXTERNAL DATA]" prefix, best-effort annotation). 4-tier response per CQ-4: CLEAN, ANNOTATED, REDACTED, QUARANTINED.

**SAD[5]: Audit Logging** | HIGH | mitigates R1,R2
- JSON-L append-only. Fields: timestamp, session_id, event_type, server_id, tool_name, action_class, params_hash (not raw), result_size, result_hash, decision, reason, score, duration_ms, model_id, turn. 30-day rotation. Async+buffered+thread-safe.

**SAD[6]: Action Gate** | HIGH | mitigates E3
- Classify tools: READ/WRITE/DESTRUCTIVE. READ: auto. WRITE: auto (configurable). DESTRUCTIVE: human confirm Y/N/always-per-tool. Pluggable callback. 60s timeout -> Denied.

**SAD[7]: Hash-Based Rug Pull Detection** | CRITICAL | mitigates T1
- SHA-256 of canonical tool definition. Registry persisted to `~/.ollama-mcp-bridge/approved_tools.json`. Check on every connect + every call. Mismatch -> BLOCK + notify + re-approve triggers re-sanitize.

**SAD[8]: Rate Limiting -- 5 Levels** | HIGH | mitigates D1,D2,D3
- max_turns (50), per_tool (20), per_server (100), temporal (30/min), result_budget (10MB). All configurable. Build implements 3/5 levels (session-level + per-server temporal). Per-tool and result-budget noted as partial.

**SAD[9]: Session Isolation** | MEDIUM | mitigates I3
- No shared mutable state. Bridge = connection scope (persistent MCP). run() = conversation scope (fresh messages + rate limiter reset). 4h idle timeout.

**SAD[10]: Trust Boundaries -- 3 Zones** | CRITICAL | mitigates all STRIDE
- Z1 UNTRUSTED (Model: intent, not action). Z2 BRIDGE (SecurityGateway: enforce, trust neither model nor tools nor results). Z3 TRUSTED (User: policy + gates + config). Every boundary crossing = enforcement point.

## Interface Contracts

**IC[1]: MCP Client <-> Security Gateway**
MCPClientManager provides ToolSchema (server, name, description, input_schema, raw_definition for hashing) to SecurityGateway. SecurityGateway OWNS MCPClientManager as constructor dependency and is its SOLE consumer. execute_tool() is the atomic operation: validate -> call -> sanitize. No bypass path.

**IC[2]: Security Gateway <-> Tool Translator**
SecurityGateway provides ApprovedTool (sanitized description, classification, definition_hash) to ToolTranslator. Translator does format conversion only -- no security decisions.

**IC[3]: Tool Translator <-> Ollama API**
ToolTranslator provides Ollama-format tools array. OllamaClient uses ONLY /api/chat endpoint. OllamaClient does NOT validate tool_call arguments.

**IC[4]: Agent Loop <-> All Layers (REVISED per DA[#6])**
AgentLoop calls SecurityGateway.execute_tool() as single atomic operation. AgentLoop NEVER calls MCPClientManager directly. Sequence: OllamaClient.chat() -> SecurityGateway.execute_tool() (internally: validate -> gate -> rate -> call -> sanitize -> audit) -> ToolTranslator.to_tool_result_message() -> loop.

**IC[5]: Library Consumer API**
Bridge is the ONLY public class. Internal classes not exported. 8 public types: Bridge, BridgeConfig, ServerConfig, SecurityConfig, BridgeResult, StreamEvent, ToolCallRecord, AuditEntry.

**IC[6]: Tool Presenter -- DEFERRED TO v1.1** (ADR[9] HATEOAS)

**SIC[1-5]: Security Interface Contracts**
- SIC[1]: Tool Ingestion -> Sanitization (MCPClientManager -> ToolSanitizer)
- SIC[2]: Model Output -> Validation (AgentLoop -> ParameterValidator)
- SIC[3]: Execution -> Audit (AgentLoop -> AuditLogger, pre+post)
- SIC[4]: Execution -> Gate (AgentLoop -> ActionGate)
- SIC[5]: Config API (Bridge -> SecurityGateway, single entry point)
