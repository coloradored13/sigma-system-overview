# BUILD SYNTHESIS: ollama-mcp-bridge Audit Remediation

## Summary

Six verified audit findings (1 P0, 2 P1, 3 P2) were remediated across five source files in ollama-mcp-bridge. The build closed a confirmed bare-filename SafePath bypass (P0 security gap), addressed dormant policy fields creating false operator confidence, added a distinct BLOCKED_PROFILE telemetry state, and implemented three lower-severity hardening items. Final result: 816 tests, 0 regressions, rubric score A- (3.75/4.0).

## Changes Implemented

### ADR[1] — Q[1] Bare filename extraction (P0) — adapters.py
**What changed**: `_extract_paths_from_args()` extended with a field-name gate. Added `_FILESYSTEM_FIELD_NAMES` frozenset (`{file, filename, path, filepath, source_file, dest_file, output_file, input_file, config_file, db, database}`) and `_is_non_path_value()` helper. When a field's key is in the frozenset, its string value is extracted as a path candidate regardless of slash prefix. Negative filter skips values matching URL/email/IP patterns.

**Key design decisions**: Layer 2 (independent basename heuristic) was dropped post-DA-challenge — field-name is the only gate, preventing false positives on generic fields like `{"input": "report.csv"}`. Negative filter inline in adapters.py rather than imported from sink_policy (avoids cross-module coupling). PathPolicy.validate_path() with allow_relative_paths=False blocks the bare names downstream — the fix enables extraction, not policy logic.

**Files**: src/ollama_mcp_bridge/adapters.py | tests/test_adapters.py (+11 tests)

### ADR[2] — Q[2] Dormant policy fields (P1) — types.py, config.py
**What changed**: `@field_validator` (Pydantic v2 mode="after") added to `DestinationPolicy` (4 validators: query_constraints, allow_redirects, allowed_methods, max_payload_bytes) and `RecipientPolicy` (1 validator: allow_first_contact). Fields annotated with `# NOT_YET_ENFORCED — raises ConfigError if set`. config.py:515 auto-conversion of allowed_outbound_domains changed from `allow_redirects=True` to `allow_redirects=False`.

**Key design decisions**: ConfigError (via ValueError) preferred over removal — breaking config load intentionally at startup rather than silently accepting a non-enforced control. This is consistent with SecurityConfig's existing `extra="forbid"` philosophy. Removing fields outright was rejected as too aggressive given potential operator configs.

**Files**: src/ollama_mcp_bridge/types.py, src/ollama_mcp_bridge/config.py | tests/test_types.py (+7 tests), tests/test_config.py (updated)

### ADR[3] — Q[3] BLOCKED_PROFILE state (P1) — types.py, security.py
**What changed**: `ToolState.BLOCKED_PROFILE` added to enum with updated lifecycle docstring. security.py:1947 (profile-block path) changed from `BLOCKED_SANITIZATION` → `BLOCKED_PROFILE`. `approve_tool()` error message updated to mention both non-approvable states. `ScanResult.blocked_profile` list was NOT added (DA compromise — see Tensions).

**Key design decisions**: ToolState split gives programmatic consumers (get_tool_states() API) the ability to distinguish poisoned tools from profile-enforcement blocks. ScanResult remains unified at blocked_sanitization to avoid API surface expansion for marginal consumer value.

**Files**: src/ollama_mcp_bridge/types.py, src/ollama_mcp_bridge/security.py | tests/test_integration.py (+4 tests)

### ADR[4] — Q[4] require_secret_scoping in validate_deployment() (P2) — security.py
**What changed**: `require_secret_scoping` assertion added to the HIGH_CONSEQUENCE validation block alongside the two existing mandatory fields (require_network_egress_controls, require_filesystem_sandbox). ConfigError message matches existing pattern. HARDENED + credential_access warning added per SQ[4b] (3-line addition to per-tool loop, caught missing in build review).

**Key design decisions**: HIGH_CONSEQUENCE mode's guarantee is that all mandatory security assertions are checked at startup — a missing assertion defeats the guarantee. Warning for HARDENED profile is advisory only (no enforcement gate).

**Files**: src/ollama_mcp_bridge/security.py | tests/test_security.py (updated + 3 new tests)

### ADR[5] — Q[5] DERIVED_EMAIL_DOMAIN in destination_influenced (P2) — sink_policy.py
**What changed**: `InfluenceType.DERIVED_EMAIL_DOMAIN` added to `destination_types` frozenset at sink_policy.py:218-221. One-line addition.

**Key design decisions**: Security-specialist confirmed enforcement path is intact — OUTBOUND classification fires via _args_contain_outbound_indicators() independently of destination_influenced. This is a telemetry-only fix for audit fidelity; severity correctly assessed as LOW.

**Files**: src/ollama_mcp_bridge/sink_policy.py | tests/test_sink_policy.py (+2 tests)

### ADR[6] — Q[6] fsync for critical audit events (P2) — audit.py
**What changed**: `import os` added. `AuditLogger.flush()` now always calls `f.flush(); os.fsync(f.fileno())` inside the with-block after the write loop. No conditional logic, no _force_sync parameter.

**Key design decisions**: Always-fsync preferred over two-path design (DA[#6] concession) — fsync overhead (~1-10ms per 10-call batch) is negligible in a security gateway context, and the always-on design eliminates the edge case where a clean session exit without triggering a critical event could lose the final buffer. close() inherits fsync durability via flush().

**Files**: src/ollama_mcp_bridge/audit.py | tests/test_audit.py (+2 tests)

## Challenge Resolutions

**DA[#1] BLOCKING — Layer 2 scope (CONCEDE)**
DA challenged that Layer 2 (independent basename heuristic) would fire on generic fields like `{"input": "report.csv"}`, causing false positives. Tech-architect conceded: Layer 2 dropped as independent gate. Field-name (Layer 1) is the only gate. Result: tighter design with better false-positive behavior. BELIEF[ADR1] raised from 0.80 to 0.88.

**DA[#5] BLOCKING — BLOCKED_PROFILE ScanResult (COMPROMISE)**
DA challenged that `ScanResult.blocked_profile` is gold-plating (AuditEventType already provides forensic distinction). Tech-architect partially conceded: ScanResult.blocked_profile list dropped; ToolState.BLOCKED_PROFILE retained. Rationale: get_tool_states() API is the programmatic consumer surface; ScanResult distinction adds API surface for marginal value. CQA later flagged this as BUG[1] CRITICAL — tech-architect correctly reclassified as INFORMATIONAL (compromise was documented and intentional).

**DA[#6] BLOCKING — always-fsync (CONCEDE)**
DA challenged the two-path _force_sync design as unnecessary complexity. Both DA and CQA agreed always-fsync is simpler and more correct — no edge case where a clean exit misses the final buffer. Tech-architect conceded. BELIEF[ADR6] raised from 0.87 to 0.91.

**CQA BUG[1] severity escalation (OVERRULED)**
CQA graded ScanResult.blocked_profile absence as CRITICAL. Tech-architect (backed by documented DA[#5] compromise) reclassified to INFORMATIONAL. ScanResult behavior is plan-compliant; the compromise is documented and the ToolState distinction is sufficient for the primary consumer use case. No fix required.

**ADR[4] SQ[4b] gap (CAUGHT IN REVIEW — FIXED)**
Build review identified that the HARDENED + credential_access warning (SQ[4b]) was not implemented. Fix applied before promotion: 3-line addition to per-tool loop + 1 test. GATE[build-fix] recorded.

## Cross-Agent Convergence

- **ADR[1] field-name gate direction**: tech-architect, security-specialist, and XVERIFY (gpt-5.4) independently converged on field-name-based detection as superior to extension-only heuristics. XVERIFY additionally confirmed the extension regex was too narrow and added the broader basename/extensionless set rationale.
- **ADR[2] ConfigError over removal**: tech-architect and security-specialist independently assessed that silent non-enforcement is worse than a breaking config error. SR[2] ("silent acceptance of unimplemented security controls is a false-confidence trap") and ADR[2] reached the same conclusion from different analytical paths.
- **ADR[5] severity as telemetry-only**: both security-specialist (via full sink classification path analysis) and tech-architect assessed DERIVED_EMAIL_DOMAIN as a telemetry gap, not an enforcement gap. Independent path tracing reached the same conclusion.
- **ADR[6] always-fsync**: DA, CQA, and tech-architect all reached the same conclusion independently. Two-path design was dropped unanimously.

## Tensions

**BLOCKED_PROFILE ScanResult — CQA vs. tech-architect**
CQA graded absence of ScanResult.blocked_profile as CRITICAL BUG. Tech-architect (plan-track) overruled, citing the documented DA[#5] compromise. Resolution: tech-architect prevailed on scope grounds — the compromise was intentional and explicitly documented in the plan. CQA's factual finding was accurate; the severity assessment was not.

**Layer 2 independent heuristic — tech-architect vs. DA**
Initial ADR[1] included Layer 2 as an independent signal. DA challenged this as creating unacceptable false positive exposure. Tech-architect conceded with evidence. Resolution: DA prevailed. Layer 2 is now dependent on Layer 1, not independent.

**fsync two-path vs. always-on — tech-architect vs. DA+CQA**
Tech-architect initially designed a _force_sync parameter for selective fsync. DA and CQA both challenged this as unnecessary complexity. Tech-architect conceded. Resolution: DA+CQA prevailed. Always-on is simpler, more correct, and eliminates an edge case.

## Known Limitations

- **ScanResult.blocked_profile absent** (known wart): ScanResult.blocked_sanitization still receives profile-blocked tools. Callers needing the distinction use get_tool_states() API or audit log. Follow-up if operators report confusion.
- **List-item bare filenames** (future ADR): adapters.py line 105-107 list branch doesn't apply field-name gate. A tool with `{"files": ["db.sqlite", "secrets.txt"]}` would not have bare filenames extracted from the list.
- **Non-listed field names**: Fields like `{"source": "db.sqlite"}` where "source" is not in _FILESYSTEM_FIELD_NAMES are not caught. The frozenset can be extended as new MCP tool patterns emerge.
- **SQ[4b] test naming** (DA[#9]): test_whitespace_value_not_extracted is a misnomer — whitespace in filesystem field IS extracted. Rename deferred.
- **HARDENED + credential_access warning** (COVERAGE[2]): SQ[4b] was implemented but the warning-path test is minimal. No enforcement consequence.
- **hardening-spec.md documentation update**: Out-of-scope for this build; dormant field documentation in hardening-spec.md needs update to reflect ConfigError behavior.
