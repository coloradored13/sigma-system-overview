#### ADR[1] — Q[1] Bare filename path extraction (P0) — XVERIFY-CORRECTED
**Decision**: Extend `_extract_paths_from_args()` with a two-layer bare-filename detector:
- **Layer 1 (field-name gate, primary)**: if field name is in `_FILESYSTEM_FIELD_NAMES` frozenset (`{file, filename, path, filepath, source_file, dest_file, output_file, input_file, config_file, db, database}`), treat the string value as a path candidate regardless of format
- **Layer 2 (basename heuristic, secondary)**: value matches `_BARE_NAME_PATTERN` — a basename-like string: non-empty, no whitespace, no `://`, no leading `http`/`https`, and EITHER contains a dot-separated extension OR is in a known extensionless filename set (`{".env", "Makefile", "Dockerfile", "README", "Jenkinsfile"}`) OR starts with a dot (hidden files)
- **Negative filter (mandatory)**: skip any value already matched as a URL, email, or IP by `_extract_values_from_args` — prevents double-reporting and false positives on destination-field URLs

**False positive analysis**:
- `{"format": "json"}` → Layer 1: `format` not in `_FILESYSTEM_FIELD_NAMES`. Layer 2: `json` has no dot, not in extensionless set → no match. SAFE.
- `{"destination": "https://api.example.com"}` → Negative filter fires (URL match). SAFE.
- `{"filename": "db.sqlite"}` → Layer 1 fires. Path candidate extracted. SafePath checks against PathPolicy. CORRECT.
- `{"content": "hello world"}` → Layer 1: not in set. Layer 2: has whitespace → no match. SAFE.
- `{"db": "prod.db"}` → Layer 1 fires (`db` in set). CORRECT.

**Implementation note**: The value extracted from Layer 1/2 must be passed to SafePath as-is. SafePath's PathPolicy.validate_path() already has `allow_relative_paths=False` by default — bare filenames (which are relative) will be BLOCKED by the policy. The fix enables the policy to actually see them.

IC[1]: `_extract_paths_from_args(args) -> list[tuple[str, str]]` signature unchanged. Internal logic extended. Callers unaffected.

SQ[1a]: Add `_FILESYSTEM_FIELD_NAMES: frozenset[str]` constant in adapters.py near `_PATH_PATTERN`

## DA Challenge Round 1 — tech-architect responses

### BLOCKING[1]: DA[#1] Layer 2 scope — concede with evidence

**DA[#1]: CONCEDE**

DA is correct. Layer 2 firing independently of Layer 1 creates unacceptable FP exposure.

Evidence: `{"input": "report.csv"}` — `input` is a plausible field name for a filename, but it is also a plausible field for a label, template name, or user-provided string. Firing SafePath on it based on extension pattern alone means any tool with an `input` field taking a `.csv` or `.txt` value gets path-validated — this will generate errors for tools that legitimately take non-path string values with extension-like endings.

**Revised ADR[1] logic**: Layer 1 (field-name) is the only gate. Layer 2 (basename heuristic) fires ONLY when Layer 1 has already matched. The purpose of Layer 2 is to expand WHAT is extracted from a field-name-gated field (to cover `.env`, `Makefile`, extension-less filenames) — not to independently flag all fields with extension-like values.

**Revised extraction logic**:
```
for each field in args:
  if field_name in _FILESYSTEM_FIELD_NAMES:
    # Layer 1 matched — extract value regardless of format
    # (slash-prefix already covered by existing logic)
    if value is a non-empty string AND not URL/email/IP:
      extract as path candidate
  # Layer 2 does NOT run independently
```

This means `{"input": "report.csv"}` → NOT extracted (input not in set). `{"filename": "report.csv"}` → extracted (filename in set). `{"path": "README"}` → extracted (path in set, extensionless OK because Layer 1 is the gate not the pattern). 

**Security coverage**: The P0 attack scenarios S1 and S2 both use field names that ARE semantically clear (`path`, `filename`, `db`, `config`). Attackers constructing `read_file(path="db.sqlite")` use the standard MCP tool parameter names — which will be in `_FILESYSTEM_FIELD_NAMES`. The narrow-field-name approach catches the actual attack surface without the FP penalty.

**Updated SQ[1b] / SQ[1c]**: Remove `_BARE_NAME_PATTERN` regex (no longer needed as independent gate). Layer 2 becomes: "when field name matched, extract value even if it has no slash." The negative filter (no URL/email/IP) is retained.

### BLOCKING[3]: DA[#6] always-fsync — concede

**DA[#6]: CONCEDE**

DA and CQA are correct. Two-path complexity is not justified.

**Evidence**: `flush()` is called in three contexts: (a) batch limit=10, (b) immediate on critical events, (c) `close()` at shutdown. All three benefit from fsync in a security audit context:
- Batch of 10: ~10 tool calls. In a security gateway this is infrequent. fsync overhead (~1-10ms) per 10 calls is negligible.
- Critical events: already identified as needing fsync.
- `close()`: process exit — must fsync to guarantee all buffered events hit disk.

CQA point about `close()` calling `flush()` is correct and strengthens the always-fsync case: if `close()` calls `flush()`, and `flush()` conditionally fsyncs only for critical events, then a session that ends cleanly without triggering a critical event in the final batch could lose the last N buffered entries. Always-fsync eliminates this edge case.

**Revised ADR[6]**: `flush()` always calls `f.flush(); os.fsync(f.fileno())`. No `_force_sync` parameter. No conditional logic. `log()` unchanged (already calls `flush()` for critical events and batch limit). `close()` unchanged.

**Updated SQ[6b]**: Remove `_force_sync` param. `flush()` always fsyncs. **Updated SQ[6c]**: No change to `log()` call sites needed.

## threat-model

### Q[1] P0 CRITICAL — Bare filename bypass in SafePath |XVERIFY:PARTIAL|source:external-openai-gpt-5.4|

**Attack surface:** `_extract_paths_from_args()` at adapters.py:50 extracts only paths matching `_PATH_PATTERN` (requires slash) or `startswith(("/","./","../","~/"))`. Bare filenames like `db.sqlite`, `secrets.txt`, `.env`, `config.json` return no entries — SafePath returns [] and the call proceeds unrestricted.

**Concrete attack scenarios:**
- S1(HIGH): Model calls `read_file(path="db.sqlite")` → SafePath returns [], MCP server resolves relative to its cwd → operator believes PathPolicy enforces containment but it doesn't
- S2(HIGH): Model calls `write_file(filename=".env", content=<attacker_payload>)` → bypasses extension allowlist, write_only, delete_allowed checks in PathPolicy entirely  
- S3(MEDIUM): Multi-hop: untrusted tool result returns "db.sqlite" → taint tracker records it → model calls `read_file(path="db.sqlite")` → sink policy fires on taint → BUT SafePath never validates the path itself
- S4(LOW-MEDIUM per XVERIFY): "arbitrary files" claim is too broad — actual impact depends on MCP server's cwd and how it resolves bare names. XVERIFY(gpt-5.4) rates this as a validation bypass for relative-bare-path arguments, not a full traversal primitive.

**Severity:** HIGH (not CRITICAL regrade). The bypass is confirmed real. Impact scope is tool-dependent — MCP servers that prefix their cwd to bare filenames narrow the blast radius; servers that pass bare names to OS directly (open("db.sqlite")) access files relative to process cwd. PathPolicy.validate_path() WOULD reject these if reached (allow_relative_paths=False by default). Bug is in extraction coverage, not in PolicyMatch logic.

**Fix direction:** Extension-based heuristic OR field-name-based heuristic (field named "path", "file", "filename", "filepath") OR both. H[1] challenge is real — "report" vs "report.txt" boundary is ambiguous. Extension + field-name combination reduces false positives.

### Q[2] P1 — Dormant policy fields creating false security confidence

**Fields in question:** `allow_first_contact` (RecipientPolicy), `query_constraints` (DestinationPolicy), `allowed_methods` (DestinationPolicy), `max_payload_bytes` (DestinationPolicy), `allow_redirects` (DestinationPolicy)

**Confirmed unenforced after code review:**
- `RecipientPolicy.allow_first_contact`: declared in types.py:453, no enforcement in `validate_recipient()` — the field exists but the check `if not self.allow_first_contact: block_new_recipients` is never executed
- `DestinationPolicy.query_constraints`: declared in types.py:613, `DestinationPolicy.matches()` validates scheme/host/port/path only — query string is never checked
- `DestinationPolicy.allowed_methods`: declared in types.py:618, never checked anywhere — HTTP method restriction is illusory
- `DestinationPolicy.max_payload_bytes`: declared in types.py:619, never enforced on outbound payload
- `DestinationPolicy.allow_redirects`: declared in types.py:617, never enforced (bridge doesn't follow redirects itself, but declaring it implies redirect controls)

**Security risk of unenforced fields:** An operator configuring `query_constraints={"token": "abc123"}` believes outbound requests are constrained to that query parameter. They are not. An operator setting `allowed_methods=["GET"]` believes POST is blocked. It is not. This is a false security confidence trap — the config surface implies enforcement that doesn't exist.

**Risk of H[2] path (remove fields):** Breaking change for any config declaring these. `extra="forbid"` on all models means adding them back later is safe, but removing them silently discards operator intent.

**Security verdict:** Keeping unenforced fields is WORSE than removing them. A field that silently accepts a value creates an exploitable trust gap — the operator believes a control is active. Recommended: remove OR add `NotImplementedError` at config load time with clear message: "query_constraints declared but not enforced — field rejected to prevent false security confidence."

### Q[3] P1 — BLOCKED_PROFILE vs BLOCKED_SANITIZATION conflation

**Confirmed in security.py:1947-1958:** Profile-block path sets `ToolState.BLOCKED_SANITIZATION` and calls `log_event(AuditEventType.TOOL_BLOCKED)`, then appends to `scan_result.blocked_sanitization`. Profile blocks (capability manifest requirements) are indistinguishable from sanitization blocks (pattern detection) in both audit logs and ScanResult.

**Security impact — can an operator miss a real security event?**
- YES: An operator monitoring `blocked_sanitization` to detect tool poisoning attempts will see profile-enforcement blocks mixed in. Signal-to-noise degrades. A real sanitization block (tool with embedded "SYSTEM: ignore instructions") could be attributed to profile enforcement or vice versa.
- YES: Forensic analysis of an incident cannot distinguish "this tool was blocked because it contained injection patterns" from "this tool was blocked because it lacked a capability manifest." These require different incident responses.
- PARTIAL: The audit log does emit `TOOL_BLOCKED` for profile failures vs `SANITIZATION_BLOCK` for pattern detection — so audit events ARE distinguishable if queried at that level. The conflation is in ToolState enum and ScanResult.blocked_sanitization list, not in AuditEventType.

**Fix direction:** Add `ToolState.BLOCKED_PROFILE`, separate `ScanResult.blocked_profile` list. H[3] warning about existing tooling parsing `BLOCKED_SANITIZATION` is valid — migration note needed.

### Q[4] P2 — require_secret_scoping missing from HIGH_CONSEQUENCE validation

**Confirmed:** `validate_deployment()` (security.py:1736-1807) checks `require_network_egress_controls` and `require_filesystem_sandbox` for `HIGH_CONSEQUENCE` mode but NOT `require_secret_scoping` (config.py:157).

**Blast radius:** An operator deploying with `deployment_mode=high_consequence` believes all required security assertions are validated at startup. `require_secret_scoping=False` (default) is silently permitted in a HIGH_CONSEQUENCE deployment. Any tool with `credential_access=True` capability can run without scoping controls.

**Concrete risk:** HIGH_CONSEQUENCE mode is explicitly for "high-stakes deployments where a single miss is bad" (config.py:50). A credential-access tool without secret scoping in that mode is exactly the failure pattern HIGH_CONSEQUENCE is meant to prevent.

**Fix:** Add `require_secret_scoping` to the HIGH_CONSEQUENCE assertion block alongside the other two mandatory fields.

### Q[5] P2 — DERIVED_EMAIL_DOMAIN missing from destination_influenced

**Confirmed in sink_policy.py:217-221:** `destination_types` set contains `{DERIVED_URL_REUSE, DERIVED_PROTOCOL_CHANGE, DERIVED_HOSTNAME_IN_URL}` but NOT `DERIVED_EMAIL_DOMAIN`. `destination_influenced` flag is False when the only taint evidence is email domain reuse.

**Can this enable data exfiltration?**
Attack chain: 
1. Web scraper tool returns HTML containing `attacker@evil.com`  
2. TaintTracker records the email value
3. Model calls a messaging tool with `to="operator@evil.com"` (same domain, different local part)
4. `_derived_match_confidence()` returns confidence=0.5, type=DERIVED_EMAIL_DOMAIN → taint IS triggered
5. BUT `destination_influenced=False` because DERIVED_EMAIL_DOMAIN is not in `destination_types`
6. `SinkPolicyEngine._classify_sink()` checks `_args_contain_outbound_indicators()` which checks for emails → returns OUTBOUND regardless
7. So the BLOCK still fires via the sink type classification path

**Net assessment:** The missing DERIVED_EMAIL_DOMAIN from `destination_influenced` does NOT create an exploitable exfil path because `SinkPolicyEngine` classifies email args as OUTBOUND via `_args_contain_outbound_indicators()` independently of `destination_influenced`. The gap is in telemetry accuracy — `destination_influenced=False` in audit logs for email-domain-taint scenarios gives operators a false picture of whether a destination was influenced. Fix is still warranted for audit fidelity, but severity is LOW (telemetry gap, not enforcement gap).

### Q[6] P2 — fsync missing on critical audit events

**Confirmed in audit.py:200-212:** `flush()` opens file in "a" mode and writes, but no `f.flush()` + `os.fsync()` call. OS page cache can hold dirty writes. On crash (power loss, OOM kill, SIGKILL), events in the page cache are lost.

**Which events are affected:** All `_CRITICAL_EVENTS` (TOOL_BLOCKED, TOOL_DENIED, TOOL_ERROR, RESULT_QUARANTINED, TAINTED_SINK_BLOCKED, TAINTED_SINK_CONFIRMED, RUG_PULL_DETECTED, RATE_LIMITED, SANITIZATION_BLOCK) immediately trigger `flush()`. These are the events where durability matters most — and these are the ones that may not survive a crash.

**Severity:** LOW-MEDIUM. Real risk only on crash during active security event. Normal session ends flush cleanly. The forensic cost is losing evidence of the last blocking event before a crash — which is exactly the event most worth preserving.

**Fix:** After `f.write(line + "\n")`, add `f.flush()` then `os.fsync(f.fileno())` for critical events only (conditional on `entry.event_type in _CRITICAL_EVENTS`). Performance impact negligible for infrequent critical events.
